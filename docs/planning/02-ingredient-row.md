# 장보기 체크리스트 재료 행(Row) — 요구사항 정의

> 작성일: 2026-07-17 · 상태: 초안 · 슬러그: 02-ingredient-row · 다음 담당: web-designer

재료 체크리스트의 **행 1개**가 담아야 할 정보·액션·상태·우선순위를 정의한다. 대상 파일은 `frontend/src/app/ingredients/IngredientsChecklist.tsx`의 `ordered.map` 내부 `<li>`. 백엔드/DB/API 변경 없음, 신규 데이터 없음.

## 1. 배경 / 문제

- 정보를 단계적으로 걷어내며(`purchaseUnit` → `purchaseLocation` 순 제거) 행에 **이름 + 필요량 + 쿠팡 아이콘**만 남았다.
- 사용자 피드백: **"쿠팡 주문 아이콘만 덜렁 넣은 게 예쁘지 않다."**
- 목표: **데이터를 되살리지 않고**, 남은 3요소의 위계·경계·라벨을 재정의해 행을 완성된 단위로 만든다. 쿠팡 링크 노출은 유지하거나 강화한다.

## 2. 현황 (코드 확인 결과)

| 항목 | 현재 구현 |
|------|-----------|
| 좌측 | 체크 원(20x20, 미체크=회색 테두리 / 체크=rose-500 채움+흰 체크) + 재료명(체크 시 `line-through` + `text-gray-400`) |
| 우측 | 필요량 `{requiredAmount}{unit}` (rose-500 bold, `min-w-[3.5rem]` 우측 정렬) + 쿠팡 링크 |
| 쿠팡 링크 | `<a target="_blank">`, 44x44 탭 타깃(`-my-2 -mr-2`), 아이콘만(장바구니 stroke, `#C0392B`), 배경 없음(hover 시 `bg-rose-50`), `aria-label="{name} 쿠팡에서 보기"`, `stopPropagation` |
| 행 | 전체가 체크 토글 (`onClick={() => toggle(...)}`, `cursor-pointer`), 체크 시 `border-rose-200 bg-rose-50` |
| 정렬 | 미체크 위 / 체크 아래 (안정 정렬) |
| 컨테이너 | `rounded-xl px-4 py-3 shadow-sm border`, 행 간 `gap-3` |

- 컴포넌트의 로컬 `Ingredient` 인터페이스는 `ingredientId / name / requiredAmount / unit / coupangUrl`만 갖는다. `purchaseUnit`·`purchaseLocation`은 **화면 계약에서 이미 제외**됨(page.tsx에서는 fetch되지만 사용 안 함). 본 문서는 이 제외를 확정으로 전제한다.
- `coupangUrl`은 타입상 `string | null` → **링크 없는 재료가 존재할 수 있음**. 현재는 링크가 없으면 아무것도 렌더하지 않아 행마다 우측 끝 위치가 달라진다.

## 3. 핵심 질문에 대한 답

### Q1. 행이 반드시 전달해야 하는 정보는?

| 정보 | 필요성 | 근거 |
|------|--------|------|
| 재료 이름 | **Must** | 마트 매대에서 찾는 검색 키. 행의 식별자. |
| 필요량(`requiredAmount`+`unit`) | **Must** | 인원 스테퍼의 결과가 드러나는 유일한 지점. 이 값이 없으면 인원 설정 기능이 무의미해진다. |
| 담음/안 담음 상태 | **Must** | 상단 진행바(`01-shopping-progress`)와 짝을 이루는 항목 단위 상태. |
| 구매 경로가 있다는 사실 + **그게 무엇인지(쿠팡)** | **Must** | 현재 "있다"만 전달되고 "무엇인지"가 전달되지 않음 → Q3 참조. |
| 그 외 | **Won't** | `purchaseUnit`·`purchaseLocation`은 사용자 결정으로 제외. API에 남은 필드는 이 둘뿐이므로 **행에 새로 추가할 데이터는 존재하지 않는다.** |

**결론: 정보는 부족하지 않다. 지금이 의도된 최소치이고, 여기에 데이터를 더할 수단도 없다.** 따라서 "앙상하다"는 인상은 **데이터 부족 문제가 아니라 구성 문제**로 다뤄야 한다. 이 문서는 데이터 추가를 해법에서 배제한다.

### Q2. 행의 액션과 우선순위

| 액션 | 성격 | 우선순위 | 근거 |
|------|------|----------|------|
| 체크 토글 | 화면의 주 목적. 재료 수만큼(평균 5~8회) 반복, 진행바·완료 상태·localStorage를 구동. 화면 내 완결. | **주(Primary)** | 이 화면의 성공 = 장보기 완주. 반복 빈도·상태 파급 모두 최대. |
| 쿠팡 열기 | 항목당 최대 1회, **외부 이탈**을 동반. 수익화 수단. | **부(Secondary), 단 축소 불가** | 빈도는 낮지만 사업 가치가 높다. "숨기지 않되 주 액션을 이기지 않는다"가 규칙. |

**규칙 (설계 원칙으로 고정)**
1. 주 액션은 **행 표면 전체**를 갖는다(현행 유지).
2. 부 액션은 **자기 경계(컨테이너)를 가진 버튼**으로 표면에서 명확히 분리된다. 경계 없는 요소가 행 위에 떠 있으면 안 된다.
3. 부 액션의 시각 무게는 주 액션의 결과물(재료명·필요량)을 **넘지 않는다**. 넘지 않되, 존재감은 텍스트 라벨로 확보한다 → **무게는 낮추고 명료도는 올린다.** 이것이 노출 유지와 미관을 동시에 만족시키는 유일한 방향이다.

### Q3. "아이콘만 있어 예쁘지 않다"의 본질 — 진단

세 가지가 겹쳤다. 우선순위 순.

| # | 원인 | 근거 | 심각도 |
|---|------|------|--------|
| 1 | **라벨 부재로 인한 의미 공백** | 장바구니 아이콘은 업계 표준 의미가 "내 장바구니"다. 여기서는 "외부 쿠팡으로 이동"이라 **관습과 충돌**한다. 텍스트 라벨 없는 아이콘은 대부분 모호하며(NN/g), 뚝딱밥에는 앱 내 장바구니가 없어 문맥으로도 해소되지 않는다. 사용자에게 "덜렁" 느껴지는 1차 원인은 **읽히지 않는 요소가 행에 놓여 있다**는 것. | 높음 |
| 2 | **시각 무게 불균형 / 경계 부재** | 44x44 탭 타깃은 `-my-2`로 행 콘텐츠 높이(≈44px) 전체를 차지한다. 즉 이 링크는 **행에서 가장 큰 요소**인데, 배경·테두리는 없고 hover에서만 나타난다. → 부 액션이 면적은 최대, 형태는 0. "떠 있는 아이콘"으로 보이는 직접 원인. 좌측(원+텍스트, 형태 있음)과 대비되어 우측만 미완성으로 읽힌다. | 높음 |
| 3 | **인접한 두 빨강의 충돌** | 필요량은 `rose-500(#f43f5e, 쿨한 핑크레드)`, 아이콘은 쿠팡 `#C0392B(웜한 벽돌레드)`. 8px 간격으로 붙어 있어 **같은 계열로 뭉뚱그려지면서 미세하게 어긋나 보인다.** 브랜드 색은 유지하되 두 요소를 색이 아닌 다른 축(경계·배경·간격)으로 분리해야 한다. | 중간 |

**본질은 위치 문제가 아니다.** 우측 배치 자체는 스캔 순서(이름 → 양 → 액션)와 일치해 타당하다. 또한 **행 전체의 정보 빈약함도 근본 원인이 아니다**(Q1). 근본은 **"의미가 없고 형태도 없는데 면적만 가장 큰 요소"가 행에 있다는 것**이다.

**해결 방향(권고)**: 쿠팡 링크를 **라벨 있는, 경계 있는, 무게는 낮은 보조 버튼**으로 바꾼다. 아이콘+"쿠팡" 텍스트로 의미를 채우고(1), 옅은 배경/테두리로 형태를 주고(2), 그 자체가 색 충돌을 완충한다(3). 부수 효과로 라벨 있는 버튼은 아이콘 단독보다 클릭률이 높은 것이 일반적 → **노출 감소 없이 수익화에 유리.**

### Q4. 체크된 재료의 쿠팡 링크

| 안 | 내용 | 평가 |
|----|------|------|
| A | 현행 그대로(무게 동일) | 이미 담은 재료에 "사러 가기"가 온전한 무게로 남아 의미 모순. 체크 행 전체가 흐려지는데 아이콘만 선명해 부조화. **기각** |
| B | **비활성 톤으로 낮추되 유지·클릭 가능** | 행의 dimmed 처리와 일관. 레이아웃 이동 없음. 마음이 바뀌면 즉시 사용 가능. **채택(권고)** |
| C | 체크 시 숨김/제거 | 리플로우 발생 + 노출 감소. 수익화 전제와 충돌. **기각** |

**정책**: 체크된 행의 쿠팡 버튼은 **저채도/저대비 톤으로 낮추고, DOM에서 유지하며, 계속 클릭 가능**하다. 근거: 해당 항목은 사용자가 이미 "담았다"고 선언 → 전환 기여 기대값이 낮고, 선명하게 남길 때의 시각적 소음 비용이 더 크다. 체크 행은 목록 하단으로 정렬되므로 주목도는 자연히 더 낮아진다(중복 억제 아님, 허용 범위).

## 4. 기능 요구사항 (MoSCoW)

| # | 기능 | 설명 | 우선순위 |
|---|------|------|----------|
| 1 | 쿠팡 액션에 텍스트 라벨 | 아이콘 단독 금지. 목적지를 알리는 짧은 라벨을 아이콘과 함께 표기(문구는 §7 결정 필요) | **Must** |
| 2 | 쿠팡 액션에 시각 경계 | 배경 또는 테두리를 가진 버튼 형태. hover에서만 나타나지 않는다(터치 기기에 hover 없음) | **Must** |
| 3 | 탭 타깃 ≥ 44x44 유지 | 라벨 추가 후에도 유지(가로로 커지므로 자연 충족) | **Must** |
| 4 | 주/부 위계 확립 | 재료명·필요량이 쿠팡 버튼보다 시각적으로 우세 | **Must** |
| 5 | 체크 시 쿠팡 버튼 dimmed 유지 | Q4-B. 클릭 가능·레이아웃 고정 | **Must** |
| 6 | `coupangUrl === null` 대응 | 버튼 자리를 **비워 두되 폭을 예약**해 필요량 우측 정렬 열을 유지 | **Should** |
| 7 | 외부 이동 명시 | `target="_blank"` + `rel="noopener noreferrer"` + aria-label 유지 | **Must** |
| 8 | 체크 상태 전달 강화 | 현행(원 채움 + line-through + 배경 rose-50) 유지 | **Must** |
| 9 | 색 충돌 완화 | 필요량 rose-500과 쿠팡 브랜드 레드가 인접해 뭉치지 않게 분리(색 외 축 사용) | **Should** |
| 10 | 좁은 화면 대응 | 360px 폭 + 긴 재료명(예: "청양고추(다진 것)")에서 줄바꿈·잘림 규칙 정의 | **Should** |
| 11 | 재료 데이터 추가 | `purchaseUnit`·`purchaseLocation` 등 정보 복원 | **Won't** |
| 12 | 쿠팡 링크 하단 모아두기 | 앞서 검토 후 노출 감소 우려로 기각된 안 | **Won't** |

## 5. 범위

**In**: `IngredientsChecklist.tsx` 행 렌더링의 정보 위계·라벨·버튼 형태·상태별 처리.
**Out**: 백엔드/API/DB, 진행바·상단 헤딩·인원 스테퍼·목록 복사/공유/초기화, 정렬 규칙, 파트너스 고지 문구 위치(현행 리스트 하단 중앙 유지), 색상값·타이포 등 비주얼 확정(→ web-designer).

## 6. 정책 / 예외

| 상황 | 동작 |
|------|------|
| `coupangUrl`이 null | 쿠팡 버튼 미렌더. 단 우측 정렬 열이 흔들리지 않도록 폭 예약. |
| 쿠팡 버튼 탭 | 체크 토글되지 않음(`stopPropagation` 유지). 새 탭 열림. |
| 체크된 행 | 이름 line-through+회색, 필요량 회색, 쿠팡 버튼 dimmed(클릭 가능). |
| 재료명이 긴 경우 | 이름 영역이 축소·줄바꿈되더라도 **필요량과 쿠팡 버튼은 절대 잘리지 않는다**(`shrink-0` 유지). |
| 파트너스 고지 | 법적 필수. 리스트 하단 중앙 현행 유지. 행 단위 반복 표기 없음. |
| 키보드 | 행 토글과 쿠팡 링크가 각각 도달 가능해야 한다(현재 `<li onClick>`은 키보드 도달 불가 — §7 참조). |

### 데이터 매핑

| 화면 요소 | 출처 |
|-----------|------|
| 재료명 | `GET /menus/{menuId}/ingredients` → `name` |
| 필요량 | `requiredAmount` + `unit` (서버가 `people` 반영해 계산) |
| 쿠팡 버튼 | `coupangUrl` (null 가능) |
| 체크 상태 | 클라이언트 `localStorage: ttukttak:checked:{menuId}` (24h 만료) |

## 7. 열린 결정 (사용자/디자이너 확인 필요)

1. **라벨 문구**: `쿠팡` / `쿠팡에서 보기` / `주문` 중 무엇인가?
   - 권고: **`쿠팡`**(+아이콘). 목적지를 정확히 말하고 360px에서도 안전. `주문`은 앱 내 주문으로 오해될 수 있어 비권장.
   - 확인 필요: 쿠팡 파트너스 정책상 브랜드명 텍스트 표기 제약 여부(단순 목적지 표기는 통상 허용 범위로 판단하나 미확인).
2. **키보드 접근성**: 현재 `<li onClick>`은 포커스 불가. 이번 범위에서 함께 고칠지 여부(별도 슬러그 분리도 가능). 권고: 이번에 함께.
3. **미니멀 대안**: 라벨 추가가 여전히 과하다고 느낀다면, **아이콘 유지 + 경계(옅은 배경) 부여 + 크기 축소**만으로 진단 #2·#3만 해결하는 축소안도 가능하다. 단 #1(의미 공백)은 남는다. 채택 여부 결정 필요.

## 8. 수용 기준 (Acceptance)

1. 재료 행을 처음 보는 사용자가 우측 액션이 **"쿠팡으로 이동"**임을 라벨만으로 알 수 있다(추측·hover 불필요).
2. 행에서 가장 시각적으로 우세한 요소는 **재료명과 필요량**이며, 쿠팡 버튼은 그보다 약하다.
3. 쿠팡 버튼은 hover/터치 이전에도 **경계(배경 또는 테두리)를 가진 버튼으로 보인다.**
4. 쿠팡 버튼의 탭 타깃은 44x44 이상이고, 탭 시 체크가 토글되지 않으며 새 탭으로 열린다.
5. 체크 시 쿠팡 버튼이 흐려지지만 **여전히 화면에 있고 클릭 가능**하며, 행 높이·정렬이 변하지 않는다.
6. `coupangUrl`이 없는 재료가 섞여도 필요량의 우측 정렬 열이 어긋나지 않는다.
7. 360px 폭에서 가장 긴 재료명 + 4자리 필요량("1000ml")에서도 필요량·버튼이 잘리지 않는다.
8. 파트너스 고지 문구가 리스트 하단에 계속 노출된다.
9. `purchaseUnit`·`purchaseLocation`이 화면에 나타나지 않는다.
10. 진행바·복사/공유/초기화·레시피 이동 동작에 회귀가 없다.
